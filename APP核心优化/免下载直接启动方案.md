# 盖世「免下载直接启动」具体方案

> 版本：V1.0 | 日期：2026.05.24
> 核心问题：网吧的游戏在共享磁盘上所以不用下载，手机没有共享磁盘，怎么办？

---

## 一、先说结论

网吧"免下载"的本质是**游戏文件已经在用户可触达的存储上了**。手机上要实现同样的"点击就玩"，有且只有四条路：

| 路径 | 原理 | 免下载程度 | 适用游戏 | 可行性 |
|------|------|-----------|---------|--------|
| **A. 云游戏** | 游戏在云端服务器跑，视频流推到手机 | 100%免下载 | 所有游戏 | ✅ 已有基础 |
| **B. 串流（用户自己的PC）** | 游戏在用户PC上跑，画面串流到手机 | 100%免下载到手机 | 用户PC上已有的游戏 | ✅ 可做 |
| **C. 按需加载（边玩边下）** | 只下载启动必需文件，剩余后台拉取 | 减少70-90%等待时间 | 特定适配过的游戏 | ⚠️ 技术难度极高 |
| **D. 智能预加载** | WiFi时自动下好用户可能玩的游戏 | 用户感知上"免下载" | 热门+小体积游戏 | ✅ 可做 |

**现实结论**：PC游戏5-50GB，手机不可能真正"不下载就本地运行"。但可以通过**A+B+D组合**让用户**感知上**接近网吧体验——点击就玩，不等待。

---

## 二、路径A：云游戏 = 真正的"网吧共享磁盘"

### 为什么云游戏 = 网吧

| 网吧 | 云游戏 |
|------|--------|
| 游戏装在服务器共享磁盘上 | 游戏装在云端GPU服务器上 |
| 客户端通过iSCSI读取 | 手机通过视频流接收画面 |
| 用户不用下载安装 | 用户不用下载安装 |
| 点击菜单→3秒启动 | 点击按钮→5-10秒启动 |

**云游戏就是手机上的网吧共享磁盘**。这不是比喻，是同一个架构的不同实现。

### 具体方案

#### 当前问题
盖世已有云游戏，但用户量只有本地游戏的1/121。原因：
- 收费（233免费）
- 入口不突出
- 用户不知道可以"直接玩不用下载"

#### 改造方案：把云游戏包装成"免下载秒玩"

**产品逻辑变更**：

```
当前：游戏详情页 → [下载并安装(本地)] [云游戏(付费)]
                    主按钮               次要入口

改为：游戏详情页 → [秒玩] [下载到本地]
                  主按钮    次要选项

"秒玩" = 云游戏，但不叫"云游戏"，叫"秒玩"
用户心智：不是"我在用云游戏服务"，而是"这个游戏可以直接玩"
```

**界面设计**：

```
┌─ 游戏详情：巫师3 ─────────────────────────────┐
│                                                │
│  [封面图]                                      │
│                                                │
│  巫师3：狂猎                                    │
│  ⭐4.7 | 42GB | RPG | 兼容性：优               │
│                                                │
│  ┌────────────────────────────────────────┐   │
│  │        🎮 秒玩（免下载）                  │   │  ← 主按钮，最大最醒目
│  │        预计5秒启动                        │   │
│  └────────────────────────────────────────┘   │
│                                                │
│  ┌──────────────────┐ ┌──────────────────┐   │
│  │ 📥 下载到本地      │ │ 🖥️ 从我的PC串流   │   │  ← 次要选项
│  │ 42GB / 约45分钟   │ │ 需PC在线          │   │
│  └──────────────────┘ └──────────────────┘   │
│                                                │
│  ── 三种方式对比 ──                             │
│  秒玩：不占手机空间，需网络，VIP免费/非VIP ¥3/时  │
│  下载：占42GB空间，离线可玩，配置后一键启动       │
│  串流：不占空间，需PC开机，画质最高              │
│                                                │
└────────────────────────────────────────────────┘
```

**定价策略（对标网吧）**：

| 方案 | 价格 | 网吧类比 | 目的 |
|------|------|---------|------|
| 新用户首次 | 免费体验15分钟 | 网吧开业优惠 | 让用户体验到"点击就玩"的感觉 |
| VIP会员 | 月费内包含X小时云游戏 | 网吧会员卡 | 绑定长期用户 |
| 按时计费 | ¥2-3/小时 | 网吧上网费 | 非会员也能用 |
| 手柄用户专属 | 买盖世手柄送X小时 | — | 硬件拉动软件 |

### 成本与可行性

| 项目 | 现状 | 需要做的 | 投入 |
|------|------|---------|------|
| 云游戏基础设施 | 已有 | 扩容+优化延迟 | 运维+机器成本 |
| 产品改造 | 入口不突出 | "秒玩"按钮+包装+定价体系 | 前端1人月 |
| 游戏库覆盖 | 部分游戏 | 扩充到Top 200 | 运维持续 |
| 延迟优化 | 未知 | 加CDN节点/优化编码 | 后端2人月 |

## 三、路径B：串流 = 用户自己的PC就是"网吧服务器"

### 原理

用户已经在PC上装了游戏（Steam库里有）。手机不用再下载，直接串流PC画面到手机。

```
用户PC（已装游戏）          用户手机（盖世APP）
┌─────────────┐            ┌─────────────┐
│ Steam库     │            │ 盖世APP     │
│ 巫师3(42GB) │──画面串流──→│ 显示画面    │
│ GTA V(100GB)│←─操作指令──│ 发送输入    │
│ ...         │            │             │
└─────────────┘            └─────────────┘
     ↑                          ↑
  游戏已装好                   不用下载
  配置已调好                   不用配置
  = 用户自己的"网吧服务器"      = 网吧客户端
```

### 为什么这对盖世特别有意义

盖世的用户画像：**手机玩PC游戏的人** → 大概率家里有PC → PC上大概率有Steam/Epic

这意味着用户的游戏**已经下载安装好了**，只是不在手机上。串流就是把PC变成用户的私人"网吧服务器"。

### 具体方案

#### 方案B-1：盖世内置串流（自研/集成）

```
首次设置（一次性）：
1. PC端安装"盖世串流助手"（轻量客户端，<50MB）
2. 登录同一个盖世账号
3. 手机APP自动发现同局域网/同账号的PC

日常使用：
1. 打开盖世APP → 游戏库自动显示"PC上的游戏"（从Steam库同步列表）
2. 点击游戏 → "从我的PC启动" 
3. PC自动启动游戏 → 画面串流到手机
4. 盖世手柄 / 虚拟按键控制

用户感知：和网吧一样——点击就玩，不用下载
```

#### 方案B-2：集成现有串流方案（Moonlight/Sunshine）

不自研串流协议，集成开源方案：
- PC端：推荐用户安装Sunshine（开源串流服务端）
- 手机端：盖世APP内嵌Moonlight客户端
- 盖世负责：游戏库展示 + 一键启动 + 手柄/按键映射

| 对比 | 自研 | 集成Moonlight/Sunshine |
|------|------|---------------------|
| 开发成本 | 高（6+人月） | 低（2人月集成） |
| 画质/延迟 | 需要时间打磨 | 已经很成熟 |
| 控制力 | 完全自主 | 受限于开源项目 |
| 建议 | 长期方向 | **短期方案，先用起来** |

#### 串流的网吧体验设计

```
┌─ 盖世游戏库 ────────────────────────────────────┐
│                                                  │
│  ┌─ 分类Tab ─────────────────────────────────┐  │
│  │ [全部] [手机可玩] [我的PC✓] [云游戏]       │  │
│  └────────────────────────────────────────────┘  │
│                                                  │
│  选中"我的PC" → 直接展示用户PC上已安装的游戏       │
│  这些游戏全部是"免下载、点击就玩"的                │
│                                                  │
│  ┌────────┐ ┌────────┐ ┌────────┐ ┌────────┐  │
│  │ 巫师3  │ │ GTA V  │ │ 原神   │ │ CS2    │  │
│  │        │ │        │ │        │ │        │  │
│  │ PC已装  │ │ PC已装  │ │ PC已装  │ │ PC已装  │  │
│  │ 🖥️串流  │ │ 🖥️串流  │ │ 🖥️串流  │ │ 🖥️串流  │  │
│  │        │ │        │ │        │ │        │  │
│  │[直接玩] │ │[直接玩] │ │[直接玩] │ │[直接玩] │  │
│  └────────┘ └────────┘ └────────┘ └────────┘  │
│                                                  │
│  💡 游戏在你的PC上运行，画面串流到手机              │
│     需要：PC开机 + 同一WiFi或远程连接              │
│                                                  │
└──────────────────────────────────────────────────┘
```

### 串流的限制和解法

| 限制 | 影响 | 解法 |
|------|------|------|
| 需要PC开机 | 不在家时不能用 | 支持远程唤醒(WoL)；提示"不在家？试试秒玩(云游戏)" |
| 需要好网络 | 外出4G/5G延迟高 | 同WiFi优先；4G下降画质+提示延迟预期 |
| PC性能要求 | 用户PC太老跑不动 | 与云游戏联动："PC跑不动？试试秒玩" |
| 首次设置 | PC要装客户端 | 极简安装引导，一个exe搞定 |

## 四、路径C：按需加载（边玩边下）— 技术探索

### 原理

不下载完整42GB，只下载启动所需的最小文件集（可能2-5GB），游戏启动后后台继续下载。

这是Xbox/PS5的"Ready to Start"功能在手机上的移植。

### 技术挑战（为什么极难）

```
PC游戏文件结构（举例：巫师3）：
├── bin/              ← 启动必需（500MB）
│   ├── witcher3.exe
│   └── 各种dll
├── content/          ← 游戏主体（40GB）
│   ├── blob0.bundle  ← 关卡数据、模型、纹理
│   ├── blob1.bundle
│   └── ...
├── movies/           ← 过场动画（3GB）
└── DLC/              ← DLC内容

问题：PC游戏不是为"按需加载"设计的。
游戏启动时会检查所有文件完整性 → 文件不全就报错/崩溃。
不像手游可以"下载主包→进入游戏→后台下载资源包"。
```

### 可行的变体方案

#### C-1：预分析+最小启动包

```
盖世后台预先分析每个游戏：
1. 监控游戏启动过程中实际读取了哪些文件（用Wine的文件访问日志）
2. 打包"最小启动集"（只包含启动到主菜单所需文件）
3. 用户下载最小启动集（通常是完整体积的10-20%）→ 进入游戏主菜单
4. 后台继续下载剩余文件
5. 用户开始新游戏/读档时，检查该关卡文件是否已下载完成

流程：
用户点击"快速启动" → 下载最小启动集(4GB/约5分钟) → 进入游戏主菜单 → 
"第一章已就绪，开始游戏" → 后台继续下载第二章...
```

| 优点 | 缺点 |
|------|------|
| 等待时间从45分钟降到5分钟 | 需要逐游戏分析文件依赖 |
| 用户可以更快开始玩 | 有些游戏启动就要读全部文件 |
| 不改变游戏本身 | 维护成本高（游戏更新后要重新分析） |

#### C-2：虚拟文件系统（FUSE）

```
更激进的方案：
1. 在Wine层面挂载一个虚拟文件系统
2. 游戏读取文件时 → 检查本地是否有 → 没有则从CDN实时拉取
3. 预测性预拉取：根据游戏进度预测接下来需要的文件，提前下载

技术栈：
FUSE (Filesystem in Userspace) → Wine → 游戏
                ↓
      缓存层（已下载的文件）
                ↓
         CDN（未下载的文件）
```

| 优点 | 缺点 |
|------|------|
| 理论上可以做到"零等待启动" | 技术复杂度极高 |
| 对游戏完全透明 | 网络延迟导致游戏卡顿/加载变慢 |
| | Android上FUSE性能开销大 |
| | 需要稳定网络，离线不可用 |

#### C路径结论

| 方案 | 可行性 | 建议 |
|------|--------|------|
| C-1 最小启动包 | 中等，可对Top 20游戏做 | **第二阶段探索**，选5个游戏试点 |
| C-2 虚拟文件系统 | 低，工程量巨大 | **暂不投入**，关注行业进展 |

## 五、路径D：智能预加载 = "网吧凌晨自动更新"

### 原理

网吧在凌晨用户不在时自动下载/更新游戏。盖世在用户WiFi+充电时自动预下载用户可能想玩的游戏。

用户第二天打开APP → 游戏已经下好了 → 感知上"不用下载，直接玩"。

### 具体方案

#### D-1：推荐预下载

```
触发条件：
- 手机连WiFi + 电量>50% + 存储剩余>10GB + 凌晨2-6点

预下载什么：
1. 用户浏览过详情页但没下载的游戏（兴趣信号最强）
2. 用户已玩游戏的更新包
3. 本周热门Top 5中用户机型兼容的（体积<2GB的优先）

预下载流程：
WiFi+充电 → 检查推荐列表 → 按优先级下载 → 下载完成后推送通知：
"🎮 巫师3已为你准备就绪，点击开始游戏"
```

#### D-2：小游戏"秒装库"

把体积小（<500MB）且兼容性好的游戏打包成"秒装库"：

```
┌─ 秒装游戏（30秒内可玩）──────────────────────────┐
│                                                    │
│  这些游戏体积小，点击后30秒内即可启动               │
│                                                    │
│  ┌────────┐ ┌────────┐ ┌────────┐ ┌────────┐    │
│  │空洞骑士 │ │星露谷   │ │传说之下 │ │铲子骑士 │    │
│  │ 420MB  │ │ 580MB  │ │ 200MB  │ │ 310MB  │    │
│  │ ✅ 99% │ │ ✅ 99% │ │ ✅ 99% │ │ ✅ 98% │    │
│  │        │ │        │ │        │ │        │    │
│  │[秒玩⚡] │ │[秒玩⚡] │ │[秒玩⚡] │ │[秒玩⚡] │    │
│  └────────┘ └────────┘ └────────┘ └────────┘    │
│                                                    │
│  点击"秒玩"：下载(30s) → 自动配置(2s) → 启动游戏    │
│  全程 < 1分钟，接近网吧体验                         │
└────────────────────────────────────────────────────┘
```

#### D-3：游戏资源P2P共享加速

模仿网吧的"三层CDN分发"：

```
网吧三层分发：
  游戏厂商CDN → 网吧本地服务器 → 客户端（LAN极快）

盖世P2P加速：
  盖世CDN → 附近已下载的用户 → 当前用户（P2P加速）
  
场景：宿舍里3个人都用盖世
  第1个人下载巫师3(42GB) → 正常CDN速度
  第2个人下载巫师3 → 从第1个人手机+CDN同时拉取 → 快2-3倍
  第3个人下载巫师3 → 从第1+2个人+CDN拉取 → 更快

这就是手机版的"网吧LAN分发"
```

## 六、组合方案——"三级火箭"

单靠一条路径不够。组合使用才能覆盖所有场景：

```
┌───────────────────────────────────────────────────────┐
│                                                         │
│  用户打开盖世APP → 看到一个游戏 → 想玩                   │
│                                                         │
│  平台自动判断最优路径：                                   │
│                                                         │
│  ├─ 手机上已有？ → 一键启动（预配置包加载）              │
│  │                                                      │
│  ├─ PC上已有？ → "从我的PC串流"（路径B）                 │
│  │   └─ PC不在线？ → 转云游戏                            │
│  │                                                      │
│  ├─ 手机上没有，但体积小(<1GB)？                         │
│  │   → "秒装：预计30秒下载完成，直接玩"（路径D-2）       │
│  │                                                      │
│  ├─ 手机上没有，体积大(>5GB)？                           │
│  │   → 主推"秒玩(云游戏)"（路径A）                       │
│  │   → 次要"下载到本地(约X分钟)"                         │
│  │   → WiFi时"已加入预下载队列，明早可玩"（路径D-1）     │
│  │                                                      │
│  └─ 以上都不行？                                        │
│      → "此游戏暂不支持，已提交适配需求"                   │
│                                                         │
└───────────────────────────────────────────────────────┘
```

### 用户感知上的"网吧体验"

通过组合方案，用户看到的是：

| 场景 | 用户操作 | 等待时间 | 和网吧对比 |
|------|---------|---------|-----------|
| 游戏已在手机上 | 点击→启动 | 3-10秒 | = 网吧 |
| 游戏在用户PC上 | 点击"从PC串流" | 5-15秒 | ≈ 网吧 |
| 小游戏(<1GB) | 点击"秒装" | 30-60秒 | ≈ 网吧 |
| 大游戏+有网 | 点击"秒玩(云)" | 5-10秒 | = 网吧 |
| WiFi预下载完成 | 点击→启动 | 3-10秒 | = 网吧 |
| 大游戏+无预下载+无网 | 必须等下载 | 30-60分钟 | ❌ 做不到 |

**最后一行是唯一做不到的场景**：大游戏+没预下载+没网络。这种情况下，任何方案都无法免下载。但通过A+B+D组合，这种场景的发生概率会大幅降低。

## 七、各路径优先级和执行排期

| 优先级 | 路径 | 投入 | 效果 | 排期 |
|--------|------|------|------|------|
| **P0** | A-改造：云游戏包装成"秒玩" | 前端1人月 + 定价策略 | 所有游戏立刻获得"免下载"入口 | **立刻做** |
| **P0** | D-2：秒装库（小游戏专区） | 运营选品 + 前端0.5人月 | 数十款小游戏实现"30秒可玩" | **立刻做** |
| **P1** | B：串流集成 | 客户端2人月 + PC端1人月 | 有PC的用户所有游戏免下载 | 1-2个月内 |
| **P1** | D-1：智能预下载 | 客户端1人月 + 后端1人月 | 提前准备好游戏，打开即玩 | 1-2个月内 |
| **P2** | D-3：P2P加速 | 后端2人月 + 客户端1人月 | 下载速度提升2-3倍 | 3-6个月 |

### 最快见效的组合（第一个月内可上线）

## 八、和"网吧模式可行性方案"的关系

前一份文档（网吧模式可行性方案）解决的是**"免配置"**——游戏文件已经在手机上了，怎么让用户不配置就能启动。

本文档解决的是**"免下载"**——游戏文件还不在手机上，怎么让用户不等待就能玩。

两份文档合在一起 = 完整的网吧体验：

```
网吧体验 = 免下载 + 免配置 + 点击启动

免下载（本文档）：云游戏秒玩 / PC串流 / 秒装库 / 智能预下载
免配置（上一份文档）：预配置包库 / 数据飞轮 / 一键启动引擎
点击启动（两份文档共同）：统一启动器，一个按钮搞定

三者合一 = "手机上的个人网吧"
```

## 九、对盖世最关键的改变

不是技术方案，而是**产品思维的转变**：

| 当前思维 | 网吧思维 |
|---------|---------|
| "用户下载游戏→配置→启动" | "用户想玩→平台想办法让他玩到" |
| 下载是用户的事 | 下载是平台的事（预下载/云游戏/串流替代） |
| 配置是用户的事 | 配置是平台的事（预配置包） |
| 失败是用户的问题 | 失败是平台的问题（自愈链/自动切换方案） |
| APP是工具 | APP是服务 |

网吧老板不会让顾客自己装游戏、自己调配置。**盖世也不应该。**

## 十一、路径B详细实施方案——PC串流

### 11.1 为什么串流对盖世是必做的

盖世用户画像 = 手机玩PC游戏的人 → **家里大概率有PC** → PC上大概率有Steam

这些用户PC上已经有几十上百个游戏，**全部已下载已安装已配置好**。串流 = 把这些游戏零成本搬到手机上。

对用户而言：不用在手机上再下载42GB，不用再配置Wine参数，PC上怎么玩手机上就怎么玩。

### 11.2 技术选型

| 方案 | 优势 | 劣势 | 建议 |
|------|------|------|------|
| **集成Moonlight+Sunshine（开源）** | 成熟、低延迟（1-5ms局域网）、NVIDIA/AMD/Intel全兼容、免费 | 需要用户PC安装Sunshine | ✅ **首选，短期方案** |
| 自研串流协议 | 完全可控、可深度集成 | 开发周期6+人月、需要大量优化 | ❌ 暂不考虑 |
| 集成Steam Link SDK | 只支持Steam游戏、Valve官方SDK | 不支持Epic/本地游戏 | ⚠️ 可作为补充 |
| Parsec/Rainway等商业方案 | 质量高 | 收费、受限于第三方 | ❌ 不适合 |

#### 预下载优先级算法

```

- 同类型推荐（用户玩过类似游戏） → 权重 1.0

可行性权重：
- 体积 < 500MB → 权重 3.0（秒装级）
- 体积 500MB-2GB → 权重 2.0（几分钟可装）
- 体积 2-10GB → 权重 1.0（WiFi下可接受）
- 体积 > 10GB → 权重 0.3（仅在明确兴趣时预下载）
- 兼容性 > 95% → 权重 ×1.5
- 兼容性 < 80% → 权重 ×0.5

时效权重：
- 游戏刚更新/刚上新 → 权重 2.0
- 限时活动相关 → 权重 2.0
- 普通 → 权重 1.0

排序后取前3-5个游戏（不超过用户剩余空间的50%）进入预下载队列
```

#### 预下载的存储管理

```
预下载游戏的生命周期：

2. 用户7天内启动了 → 转为"已安装游戏"，正常保留
```
