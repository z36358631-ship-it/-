## 一、核心用户心智定义

### 1.1 用梁宁"需求三角"验证盖世的真需求

**需求三角 = 痛点 × 强场景 × 支付意愿**（三者缺一不成立）

| 维度 | 盖世用户的真实状态 | 验证证据 |
|------|------------------|---------|
| **痛点** | 想在手机上玩PC游戏，但启动成功率只有3.9%，96%的尝试以失败告终 | 人均尝试14.6次、配置切换27次、D1留存仅20% |
| **强场景** | 通勤/躺床/外出时，想玩PC游戏但没有电脑在身边 | 用户装多个模拟器"哪个能启动就用哪个"——行为证据 |
| **支付意愿** | 愿意花时间折腾（166秒配置+5分钟设置），说明需求是真的；部分付费（云游戏/CDKEY） | 39万用户坚持在WineActivity平均游玩43分钟 |

**结论**：需求是真的，用户用脚投票了。但产品没有兑现承诺，导致大面积流失。

### 1.2 用户心智占位

根据梁宁心智三层结构：

| 层次 | 当前状态 | 目标状态 |
|------|---------|---------|
| **品类心智** | "盖世=安卓模拟器"（和Winlator/233竞争） | **"盖世=手机上的游戏主机"** |
| **特性心智** | 无（用户说不出盖世比233好在哪） | **"点开就能玩"**——确定性 |
| **情感心智** | 挫败感（96%失败）、不信任（崩溃12.3%） | 安心（它推荐的一定能玩）、爽（手柄+手机=掌机） |

### 1.3 一句话心智

> **盖世游戏 = 手机上的游戏主机，点开就能玩**
>
> 用户向朋友推荐时应该说：
> "下载盖世，你Steam里的游戏在手机上直接玩，不用折腾配置。"

### 1.4 心智验证：用户已有的认知框架

| 用户已有心智 | 盖世应该顺应的方式 | 不应该做的 |
|------------|-----------------|-----------|
| "游戏主机开机→选游戏→按A→开始玩" | 模拟游戏主机体验：游戏库→选择→一键启动 | 让用户配置Wine版本、DXVK参数 |
| "网吧坐下就能玩" | 网吧预制盘模式：预配置好，用户只管点 | 让用户当自己的"网管" |
| "Steam库里的游戏我买了就能玩" | 绑定Steam后立刻可玩，不需要额外折腾 | 让用户手动导入、手动选配置 |
| "手柄插上就能用" | 盖世手柄连接→自动识别游戏→自动映射 | 手动配映射 |

## 二、用户之旅：强场景×强感知 需求表

### 2.1 用户旅程全景

| | APP体验前 | 游戏启动前 | 游戏体验中（前15分钟） | 游戏体验后 | 游戏体验外 |
|---|---|---|---|---|---|
| **用户场景** | 刷视频/听朋友说→搜索下载→首次打开APP | 找到想玩的游戏→获取文件→点击启动 | 进入游戏→操控→调设置→沉浸游玩 | 退出游戏→存档→第二天回来 | 推荐朋友/连接手柄/付费决策/社区 |
| **用户想法** | "装一个试试，不行就删"→"我要怎么开始？游戏在哪？能不能玩？" | "我想玩巫师3，有吗？要下多久？"→"点一下就让我玩！别让我选看不懂的东西！" | "操作怎么弄？"→"画质帧率怎么调？"→"还不错，继续玩" | "存档在哪？下次还在吗？"→"昨天能玩的今天还能玩吧？" | "推荐给朋友"→"手柄连上怎么没反应？"→"值不值得充钱？" |
| **场景强度** | ⭐⭐⭐⭐ 强（决定装不装、留不留） | ⭐⭐⭐⭐⭐ 生死级（96%用户死在这里） | ⭐⭐⭐⭐ 强（决定满意度和时长） | ⭐⭐⭐⭐ 强（决定是否回来） | ⭐⭐⭐ 中（决定传播和付费） |
| **当前产品表现** | 登录页拦住91.8%→空荡游戏库→不知所措 | 搜索无引导、卡片无兼容信息→配置页166秒27次切换→96%失败无反馈 | 虚拟按键仅3.3%覆盖→必须退出才能调设置→崩溃12.3% | 无存档管理、Prefix崩存档丢→D1留存仅20% | 无分享机制→手柄连接仅2929人→云游戏收费但没信任 |
| **产品应该提供的** | 跳过登录→直达"秒玩/秒装"→"选一个，现在就玩"→60秒内玩到第一个游戏 | 兼容性标签("一键可玩")→网吧预制盘模式(3分钟可玩)→预配置一键启动→失败自愈链 | 自动推荐虚拟按键/手柄自动映射→游戏内浮窗调设置→不崩溃稳定运行 | 自动存档备份+云同步→次日推送"一键继续"→存档跨设备可用 | P2P传游戏(分享即拉新)→手柄即插即用零配置→免费体验15分钟→建信任后VIP转化 |
| **核心指标** | 新用户60秒内玩到游戏比例；D1留存 | 启动成功率3.9%→40%+；配置页时长166s→0 | 游戏平均时长43分钟→60分钟+；崩溃率→<2% | D1留存20%→30%+；D7留存3.7%→10%+ | 手柄连接用户→1万+；VIP转化率→5%+ |

### 2.2 关键发现：五个"生死场景"

从用户旅程中提取**场景强度⭐⭐⭐⭐⭐**的环节：

| # | 生死场景 | 用户底线 | 不满足=直接卸载 |
|---|---------|---------|---------------|
| 1 | 获取游戏文件 | "不想等一小时下载" | 提供秒玩/预制盘/P2P等替代方案 |
| 2 | 启动游戏 | "点一下就让我玩" | 预配置一键启动 |
| 3 | 启动失败 | "告诉我怎么回事，帮我解决" | 自愈链+透明反馈 |
| 4 | 存档安全 | "我的进度不能丢" | 自动备份+云同步 |
| 5 | 手柄体验（手柄用户） | "我花钱买的手柄，连上就该直接用" | 即插即用零配置 |

    ↑

    │        ╱       · 游戏内实时调画质
    │      ╱         · 云预制盘3分钟可玩
    │    ╱
    │   ╱
    ├──╱─────────── ★ 期望型需求（做多得多）────────────
    │╱              · 支持的游戏数量越多越好
    │               · 启动速度越快越好
    │               · 画质/帧率越高越好
    │               · 搜索/发现体验
    │               · 社区内容丰富度

    │

    │    · 游戏能启动（成功率>80%）     ← 当前3.9% ❌致命

    ↓ 用户不满

## 四、15项功能优先级排序

### 4.1 排序原则（来自梁宁框架）

1. **先守底线**：基本型需求没满足，其他全白做（用户直接走人）
2. **强场景优先**：解决"用户不得不做"的环节，比"锦上添花"重要100倍
3. **行为验证**：用户已经在用行为证明的需求（39万人坚持玩43分钟）> 用户嘴上说想要的
4. **竞品抄不了 > 竞品能抄**：资源有限时，做差异化壁垒的事

## 六、"网吧预制盘模式"在整体策略中的定位

### 6.1 网吧预制盘 = #11"快"的具体实现

网吧预制盘模式解决的是用户旅程中**"获取游戏文件"**和**"启动游戏"**两个生死场景：

```
                              网吧预制盘模式覆盖范围
                    ┌─────────────────────────────────────┐
                    │                                       │
用户想玩游戏 → [获取文件] → [配置] → [启动] → 玩到游戏
                    │    ↑        ↑       ↑                │
                    │    │        │       │                │
                    │  最小启动包  预配置包  一键启动         │
                    │  +按需加载   自动注入  自愈链          │
                    │  +P2P传输                            │
                    └─────────────────────────────────────┘
```

### 6.2 对无PC用户的Steam游戏获取方案

| 方式 | 场景 | 用户操作 | 等待时间 |
|------|------|---------|---------|
| **秒玩（云游戏）** | 任何网络 | 点击→云端启动 | 5秒 |
| **最小启动包** | WiFi | 下载1-2GB启动包→边玩边加载 | 3-5分钟 |
| **P2P近场传输** | 朋友已有 | 从朋友手机直传 | 10-20分钟(取决于大小) |
| **VIP租号** | 没有Steam账号 | 开VIP→平台提供账号→直接玩 | 即时 |
| **WiFi预下载** | 不急 | 晚上WiFi自动下好 | 次日 |
| **传统下载（压缩加速）** | 以上都不行 | 正常下载，但压缩后体积减半 | 30-60分钟 |

## 七、硬件协同（#13）具体策略

### 7.1 互利互惠闭环设计

```
         ┌──── 拉APP用户 ────┐
         │                     │
         ▼                     │
  ┌─────────────┐      ┌──────┴──────┐
  │  盖世手柄    │      │  盖世APP     │
  │  (硬件)     │      │  (软件)      │
  └──────┬──────┘      └──────┬──────┘
         │                     │
         └──── 拉手柄销售 ───┘

具体机制：
APP → 拉动手柄销售：
  · APP内"手柄专区"展示手柄适配游戏："用手柄玩这些游戏体验提升200%"
  · 游戏内提示："检测到你在用触屏，连接盖世手柄操控更精准"
  · VIP特权："购买手柄送6个月VIP（含秒玩时长+租号）"

手柄 → 拉动APP用户：
  · 手柄包装内二维码："扫码下载盖世APP，解锁全部手柄功能"
  · 手柄开机配对时引导安装盖世APP
  · 手柄固件更新必须通过盖世APP（锁定）

差异化体验（只有盖世手柄+盖世APP才有）：
  · 即插即用零配置（第三方手柄仍需手动映射）
  · 手柄导航APP（摇杆选游戏，A键启动）
  · 背键宏（自定义连招）
  · 游戏自适应震动
  · 陀螺仪辅助瞄准
```

## 八、总结：盖世游戏要打造的核心用户心智

### 8.1 一句话

> **"盖世游戏 = 手机上的游戏主机，Steam里的游戏点开就能玩"**

### 8.2 心智验证（用梁宁"替代方案"检验法）

| 检验维度 | 回答 |
|---------|------|
| 用户现在怎么解决这个需求？ | 装Winlator/233/GameHub手动配置，或者不玩 |
| 用户的替代方案有多痛苦？ | 极痛苦：成功率<10%，配置看不懂，崩溃频繁 |
| 盖世比替代方案好在哪？ | "点开就能玩"+自有手柄+数据飞轮=确定性+便捷+差异化 |
| 用户愿意为此付多少？ | 时间（43分钟/次游戏时长）+ 金钱（VIP/租号意愿待验证） |
| 切换到盖世的成本高吗？ | 极低（下载一个APP），唯一壁垒是"这次也不行吧？"的不信任 |

### 8.3 产品打造的核心逻辑

```
第一步（Q3）：兑现承诺
  "点开就能玩" → 从口号变成现实（成功率40%+）
  让用户第一次体验就成功 → 打破"这次也不行吧"的不信任

第二步（Q4）：建立差异化认知
  "为什么用盖世不用233？" → "因为盖世点开就能玩，别的得折腾"
  + 手柄体验是独有的 → "盖世手柄+盖世APP最顺手"

第三步（Q1+）：形成壁垒
  数据飞轮 → 用的人越多越好用（配置越准）
  存档/配置沉淀 → 迁移成本
  手柄锁定 → 硬件用户跑不掉
  社区UGC → 内容壁垒

终局心智：
  "手机玩PC游戏 = 盖世" （品类心智占位）
  就像 "搜索 = 百度" / "短视频 = 抖音" / "电商 = 淘宝"
```
