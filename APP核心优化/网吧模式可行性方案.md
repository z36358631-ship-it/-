# 盖世游戏「网吧模式」可行性方案

> 版本：V1.0 | 日期：2026.05.24
> 核心命题：网吧的"点击就玩"体验，能不能搬到手机上？

---

## 一、网吧到底做了什么

用户在网吧的体验是：打开游戏菜单 → 点击图标 → 3秒后游戏启动。没有下载、没有安装、没有配置。

这背后是一套**三层架构**把所有脏活藏起来了：

```
┌─────────────────────────────────────────────────┐
│  用户看到的                                       │
│  游戏菜单 → 点击 → 玩                             │
└───────────────────┬─────────────────────────────┘
                    │ 用户感知边界
┌───────────────────▼─────────────────────────────┐
│  层1：系统虚拟盘（母盘）                           │
│  统一的 Windows + 驱动 + 运行库                    │
│  所有机器共享同一个只读镜像，重启还原               │
├─────────────────────────────────────────────────┤
│  层2：游戏虚拟盘（4-8TB共享存储）                  │
│  所有游戏预装好，客户端通过 iSCSI 直接读取         │
│  网维大师/信佑自动拉取更新（三层CDN分发）           │
├─────────────────────────────────────────────────┤
│  层3：回写盘（临时存储）                           │
│  存档、临时文件 → 重启后清除                       │
└─────────────────────────────────────────────────┘
```

### 网吧能做到"点击就玩"的三个前提

| 前提 | 网吧怎么满足 | 手机端的挑战 |
|------|------------|------------|
| **硬件统一** | 同批采购，一套配置通杀 | 安卓碎片化，芯片/GPU/RAM千差万别 |
| **存储集中** | 游戏在服务器虚拟盘，不在本地 | 手机本地存储有限，网络不稳定 |
| **运维前置** | 网管/管理系统替用户做完所有配置 | 没有网管，平台必须自动化 |

**核心洞察**：网吧模式的本质不是"预装游戏"，而是**把不确定性消除在用户感知之前**。

## 二、盖世当前的问题——反网吧体验

把盖世当前体验和网吧对比：

| 环节 | 网吧体验 | 盖世当前体验 | 差距 |
|------|---------|------------|------|
| 找游戏 | 分类菜单，所有游戏都能玩 | 游戏卡片无兼容性信息，不知道能不能玩 | 大 |
| 启动 | 点击图标，3秒启动 | 配置页停留166秒，切换27次开关 | 致命 |
| 配置 | 不存在，网管已配好 | 用户手动选Wine版本/DXVK/分辨率等 | 致命 |
| 失败处理 | 几乎不会失败 | 96.1%启动失败，无反馈，盲目重试14.6次 | 致命 |
| 更新维护 | 凌晨自动更新，用户无感 | 用户手动处理 | 大 |

**盖世是"反网吧"的**：网吧消除了所有不确定性，盖世把所有不确定性都甩给了用户。

## 三、手机端"网吧模式"的四条技术路径

### 路径对比

| 路径 | 原理 | 网吧类比 | 优势 | 劣势 | 适用场景 |
|------|------|---------|------|------|---------|
| **A. 云游戏串流** | 游戏跑在云端GPU服务器，视频流推送到手机 | = 云网吧（瘦终端） | 对手机性能零要求，任何游戏都能跑 | 延迟敏感、带宽费用高、离线不可用 | 大型3A、本地跑不动的 |
| **B. 本地Wine + 预配置库** | Wine模拟器本地运行，平台下发游戏专属配置包 | = 网吧的游戏虚拟盘+预装配置 | 离线可玩、无延迟、成本低 | 受限于手机性能，仅支持中低配游戏 | 2015年前的PC游戏 |
| **C. 按需资源加载** | 游戏核心文件本地，大资源从CDN按需拉取 | = 网吧虚拟盘的读取机制 | 省存储空间 | 技术复杂度极高，Wine层面难实现 | 暂不可行 |
| **D. 混合模式（推荐）** | B为主+A兜底，平台自动判断走哪条 | = 网吧本地+云网吧混合 | 覆盖面最广，体验最接近网吧 | 需要两套基础设施 | 所有场景 |

### 推荐路径：D. 混合模式

```
用户点击"启动"
  ↓
平台自动判断
  ├── 本机能跑（配置匹配度>80%）→ 本地Wine启动，加载预配置包 → 3-10秒进入游戏
  ├── 本机勉强能跑（50-80%）→ 提示"本地模式可能卡顿，推荐云游戏" → 用户选择
  └── 本机跑不动（<50%）→ 自动切换云游戏模式 → 串流启动
  
用户感知：点击 → 玩（和网吧一样）
平台在背后：自动选择了最优运行方案
```

## 四、核心方案：预配置库系统（网吧的"游戏虚拟盘"在手机上的实现）

### 4.1 架构设计

网吧的游戏虚拟盘 = 所有游戏预装好 + 统一配置好。手机上不可能预装所有游戏，但可以**预配置好所有游戏**。

```
┌──────────────────────────────────────────────────────┐
│  云端：配置中心（= 网吧的管理服务器）                    │
│                                                        │
│  ┌──────────────┐  ┌──────────────┐  ┌──────────────┐ │
│  │ 游戏配置包库  │  │ 芯片适配矩阵  │  │ 数据飞轮引擎  │ │
│  │ 每个游戏一套  │  │ 骁龙/天玑/联发 │  │ 成功率统计    │ │
│  │ Wine版本     │  │ 科/三星Exynos │  │ 自动优化推荐  │ │
│  │ DXVK版本    │  │ GPU型号→最优  │  │              │ │
│  │ Box64参数    │  │ 驱动+参数     │  │              │ │
│  │ 分辨率/帧率  │  │              │  │              │ │
│  │ 虚拟按键方案  │  │              │  │              │ │
│  └──────────────┘  └──────────────┘  └──────────────┘ │
└────────────────────────┬─────────────────────────────┘
                         │ OTA推送/按需拉取
┌────────────────────────▼─────────────────────────────┐
│  手机端：盖世APP（= 网吧的客户端）                       │
│                                                        │
│  ┌──────────────┐  ┌──────────────┐  ┌──────────────┐ │
│  │ Wine运行环境  │  │ 本地配置缓存  │  │ 游戏启动器   │ │
│  │ （= 系统虚拟盘）│  │（= 游戏虚拟盘）│  │（= 网吧菜单）│ │
│  │ 基础运行时    │  │ 已拉取的配置包│  │ 点击→自动    │ │
│  │ 统一更新      │  │ 按游戏+机型   │  │ 加载配置→启动│ │
│  └──────────────┘  └──────────────┘  └──────────────┘ │
└──────────────────────────────────────────────────────┘
```

### 4.2 配置包结构（每个游戏 × 每类芯片 = 一个配置包）

```json
{
  "game_id": "gta_v",
  "game_name": "GTA V",
  "chipset_group": "snapdragon_8_gen3",
  "version": "2.1.0",
  "updated_at": "2026-05-20",
  
  "wine_config": {
    "wine_version": "wine-9.8-staging",
    "dxvk_version": "2.5.1",
    "box64_version": "0.3.2",
    "box64_dynarec": "full",
    "vk_driver": "turnip-mesa-24.3"
  },
  
  "display": {
    "resolution": "960x540",
    "fps_cap": 30,
    "renderer": "dxvk",
    "vsync": false
  },
  
  "controls": {
    "virtual_gamepad_scheme": "gta_v_default_v3",
    "geek_controller_mapping": "gta_v_x4_optimized",
    "touch_mode": "direct"
  },
  
  "compatibility": {
    "status": "verified",       // verified / playable / unstable / unsupported
    "success_rate": 87.3,       // 数据飞轮统计
    "tested_devices": 1247,     // 测试样本量
    "known_issues": ["opening cutscene may stutter for 5s"],
    "workarounds_applied": ["skip_intro_video=true"]
  },
  
  "size_kb": 12                 // 配置包本身极小，只是参数
}
```

**关键点**：配置包本身只有几KB（只是参数），不是游戏本体。真正的大小开销在游戏文件上，这一点和网吧不同——网吧游戏在共享盘上用户无感，手机上游戏必须下载到本地。

### 4.3 与网吧的逐项对应

| 网吧组件 | 盖世手机端对应 | 实现方式 |
|---------|-------------|---------|
| **网维大师（管理系统）** | 配置中心后台 | 后端服务：管理所有游戏配置包，按机型分发 |
| **系统虚拟盘（统一OS）** | Wine运行环境（APP内置） | Wine + Box64 + DXVK + Turnip 打包在APP内，统一更新 |
| **游戏虚拟盘（预装游戏）** | 预配置库 + 用户本地游戏文件 | 配置包云端下发（几KB），游戏文件用户自行导入或购买 |
| **三层CDN更新** | 配置包OTA推送 | 后台检测到配置更新 → WiFi时自动拉取最新配置包 |
| **还原穿透** | 运行时热更新 | 更新Wine/DXVK版本不影响用户存档和游戏文件 |
| **游戏菜单** | 盖世游戏库（带兼容性标签） | 游戏卡片显示"一键可玩"/"建议云游戏"等状态 |
| **网管** | 数据飞轮 + 人工QA | 自动收集成功/失败数据优化配置，QA团队人工验证Top游戏 |

## 五、用户体验设计——让手机变成"个人网吧"

### 5.1 游戏库 = 网吧菜单

网吧菜单的特征：**只展示能玩的游戏**。

```
┌─ 盖世游戏库 ──────────────────────────────────────────┐

│  ┌─ 筛选栏 ──────────────────────────────────────────┐ │
│  │ [全部] [一键可玩 ✓] [需要配置] [仅云游戏]          │ │
│  └────────────────────────────────────────────────────┘ │
│                                                        │
│  默认选中"一键可玩" → 只看到确定能玩的                   │
│                                                        │
│  ┌────────┐ ┌────────┐ ┌────────┐ ┌────────┐         │
│  │ GTA V  │ │ 巫师3  │ │ 空洞骑士│ │ 星露谷  │         │
│  │        │ │        │ │        │ │        │         │
│  │ ✅可玩  │ │ ✅可玩  │ │ ✅可玩  │ │ ✅可玩  │         │
│  │ 87%成功 │ │ 92%成功 │ │ 99%成功 │ │ 99%成功 │         │
│  │ 2.1GB  │ │ 3.8GB  │ │ 420MB  │ │ 580MB  │         │
│  │ ⭐4.5   │ │ ⭐4.7   │ │ ⭐4.9   │ │ ⭐4.8   │         │
│  │        │ │        │ │        │ │        │         │
│  │[一键启动]│ │[一键启动]│ │[一键启动]│ │[一键启动]│         │

│  （折叠区域，不主动展示）                                │
└────────────────────────────────────────────────────────┘
```

### 5.2 启动流程对比

**当前（反网吧）**：
```
点击启动 → 配置页（166秒）→ 选Wine版本 → 选DXVK → 选分辨率 → 
切27次开关 → 点启动 → 失败 → 不知道为什么 → 再试 → 又失败 → 
重复14.6次 → 96.1%最终放弃
```

**网吧模式**：
```
点击"一键启动" → "正在加载最优配置..."（2秒）→ 游戏启动
                                              ↓ 如果失败
                                         "正在切换方案2..."（自动）
                                              ↓ 如果还失败
                                         "本地启动困难，正在为你启动云游戏版..."
                                              ↓
                                         云游戏模式启动
```

### 5.3 三档游戏状态（对应网吧菜单的"可用/维护中/下架"）

| 状态 | 含义 | 展示方式 | 类比网吧 |
|------|------|---------|---------|
| **✅ 一键可玩** | 该游戏+该机型有验证过的配置包，成功率>80% | 绿色标签，默认展示，启动按钮高亮 | 网吧菜单里的游戏 |
| **⚠️ 可尝试** | 有配置包但成功率50-80%，或未充分验证 | 黄色标签，提示"可能需要调整" | 网吧新上架的游戏 |
| **☁️ 建议云游戏** | 本地配置成功率<50%或硬件不支持 | 蓝色标签，点击直接进入云游戏 | 网吧里"机器带不动"的游戏 |

## 六、数据飞轮——网吧有网管，盖世有数据

网吧的配置是网管手动调好的。盖世没有网管，但有312万MAU产生的数据。

### 6.1 飞轮机制

```
阶段一（冷启动）：人工QA
  产品/QA团队 → 手动测试 Top 100 游戏 × Top 10 芯片组合
  → 产出 ~1000 个配置包 → 覆盖 80% 用户的 80% 需求
  
阶段二（数据驱动）：
  用户启动游戏 → 上报：机型+游戏+配置快照+成功/失败+帧率+崩溃日志
       ↓
  后台自动分析：同机型+同游戏，哪个配置组合成功率最高？
       ↓
  自动生成/更新配置包 → 推送给同机型用户
       ↓
  新用户用了更优配置 → 成功率提升 → 上报更多成功数据 → 配置更准...

阶段三（飞轮加速）：
  数据量 → 配置覆盖率 → 成功率 → 用户量 → 更多数据
  竞品要追赶 = 需要相同规模的用户 × 相同时间的积累
```

### 6.2 冷启动策略

网吧第一天开业时，游戏也是网管手动装好的。盖世的"冷启动"也需要人工：

| 阶段 | 覆盖目标 | 方法 | 预估工作量 |
|------|---------|------|-----------|
| **第一批** | Top 50 游戏 × Top 5 芯片 = 250个配置包 | QA团队手动测试+调优 | 2人×1个月 |
| **第二批** | Top 200 游戏 × Top 10 芯片 = 2000个配置包 | 手动测试 + 社区玩家反馈 + 数据上报辅助 | 2人×2个月 |
| **自动化** | 长尾游戏 | 基于相似游戏的配置自动推荐（同引擎、同年代→相似配置） | 算法开发1人月 |

### 6.3 与竞品（GameHub/Winlator/233）的差异

| 能力 | Winlator | GameHub | 233乐园 | 盖世（网吧模式） |
|------|----------|---------|---------|----------------|
| 配置方式 | 纯手动 | 预配置（开发者维护） | 不透明 | **预配置（数据飞轮+人工QA）** |
| 配置覆盖量 | 0 | 几十个 | 未知 | 目标2000+ |
| 失败兜底 | 无 | 无 | 有云游戏 | **自动切配置B/C→云游戏** |
| 手柄集成 | 无 | 无 | 第三方 | **盖世手柄即插即用** |
| 数据飞轮 | 无 | 无 | 可能有但未公开 | **核心壁垒** |

## 七、执行方案

### 7.1 分期规划

#### 第一期：「网吧菜单」（1-2个月）——让用户看到哪些能玩

| 任务 | 具体内容 | 产出 |
|------|---------|------|
| 配置包格式定义 | 设计配置包JSON Schema，定义字段和版本管理规则 | 技术规范文档 |
| Top 50 游戏 × Top 5 芯片测试 | QA手动测试，记录最优配置参数 | 250个配置包 |
| 配置中心后台 | 配置包上传/管理/分发的后台系统 | 管理后台 |
| 游戏卡片增加兼容性标签 | "一键可玩"/"可尝试"/"建议云游戏"三档标签 | 前端改造 |
| 游戏库默认筛选"一键可玩" | 默认只展示有验证配置包的游戏 | 前端改造 |

**第一期做完后的用户体验**：打开游戏库 → 看到的游戏都标着"一键可玩" → 选一个 → 一键启动 → 大概率成功。**这就已经接近网吧体验了。**

#### 第二期：「自动配置」（2-3个月）——让点击就能玩

| 任务 | 具体内容 | 产出 |
|------|---------|------|
| 一键启动引擎 | 点击启动→自动拉取配置包→注入Wine参数→启动游戏，用户不碰配置页 | 客户端核心改造 |
| 失败自愈链 | 配置A失败→自动尝试B→尝试C→提示云游戏 | 客户端改造 |
| 数据上报系统 | 每次启动上报机型+游戏+配置+结果 | 埋点+后端 |
| 配置包自动推送 | WiFi环境下预拉取常玩游戏的配置包 | 客户端改造 |
| 扩展到Top 200 游戏 × Top 10 芯片 | 持续QA测试 + 数据上报辅助优化 | 2000个配置包 |

**第二期做完后的用户体验**：点击启动 → 自动加载最优配置 → 3-10秒进入游戏 → 失败了平台自动处理。**这就是网吧体验。**

#### 第三期：「数据飞轮」（3-6个月）——越用越好用

| 任务 | 具体内容 | 产出 |
|------|---------|------|
| 数据飞轮引擎V1 | 基于上报数据自动生成/优化配置包 | 算法服务 |
| 相似游戏推荐配置 | 同引擎同年代的游戏→自动套用相似配置 | 覆盖长尾游戏 |
| 手柄配置包集成 | 配置包增加盖世手柄映射方案，即插即用 | 手柄体验差异化 |
| 虚拟按键方案集成 | 配置包增加推荐按键布局 | 虚拟按键覆盖率提升 |
| 社区配置分享 | 用户上传自定义配置包 → 被评分 → 优质方案进入官方库 | UGC生态 |

### 7.2 成本估算

| 阶段 | 投入 | 核心产出 | 预期效果 |
|------|------|---------|---------|
| 第一期 | QA 2人×1月 + 后端1人月 + 前端1人月 = **5人月** | 250个配置包 + 游戏库标签 | 用户能看到哪些游戏确定能玩 |
| 第二期 | 客户端2人月 + 后端1人月 + QA持续 = **5人月** | 一键启动 + 失败自愈 + 数据上报 | 启动成功率 3.9% → 40%+ |
| 第三期 | 算法1人月 + 后端1人月 + 客户端1人月 = **3人月** | 数据飞轮 + 手柄/按键集成 | 长尾游戏覆盖 + 竞品壁垒 |
| **合计** | **~13人月** | | |

## 八、游戏来源与存档问题——和网吧最大的不同

### 8.1 网吧 vs 盖世的游戏来源差异

网吧的游戏是**网吧统一采购/安装**的，用户不需要管游戏从哪来。但盖世的用户游戏来源完全不同：

| 来源 | 占比（估） | 用户行为 | 盖世当前支持 |
|------|-----------|---------|------------|
| **Steam绑定** | ~60% | 绑定Steam账号，从Steam库拉取已购游戏 | ✅ 已支持 |
| **Epic绑定** | ~15% | 绑定Epic账号 | ✅ 已支持 |
| **本地导入** | ~20% | 用户从PC拷贝游戏文件到手机/从网上下载 | ✅ 已支持（58%失败率） |
| **CDKEY购买** | ~5% | 在盖世商城买Key激活 | ⚠️ 刚起步 |

**核心矛盾**：网吧统一提供游戏所以能统一配置；盖世的游戏来源分散，同一个游戏可能来自不同平台、不同版本、不同安装方式。

### 8.2 解决方案：统一游戏识别层

无论游戏从哪来，盖世需要一个**游戏识别层**来"认出"这是什么游戏，然后匹配配置包：

```
用户导入/绑定的游戏
  ↓
┌─────────────────────────────────────┐
│  游戏识别层（Game Identifier）        │
│                                       │
│  识别方式（按优先级）：                 │
│  1. Steam/Epic AppID → 直接匹配       │
│  2. 游戏exe文件名 + 版本号 → 模糊匹配  │
│  3. 游戏文件夹结构指纹 → 特征匹配      │
│  4. 用户手动确认："这是XX游戏吗？"      │
│                                       │
│  输出：game_id（唯一标识）              │
└───────────────────┬─────────────────┘
                    ↓
          配置中心匹配 game_id + chipset → 下发配置包
```

**不同来源的处理差异**：

| 游戏来源 | 识别难度 | 配置包适配难度 | 特殊处理 |
|---------|---------|-------------|---------|
| Steam绑定 | 极低（有AppID） | 低 | Steam版游戏配置最稳定 |
| Epic绑定 | 低（有AppID） | 低 | 与Steam类似 |
| 本地导入 | **高**（可能是盗版/魔改版/不同版本） | **中-高** | 需要识别游戏版本；不同版本可能需要不同配置 |
| CDKEY购买 | 极低 | 低 | 盖世全链路控制 |

**本地导入的特殊处理**：
- 本地导入的游戏版本不确定（可能是旧版、魔改版、带MOD版）
- 方案：识别后匹配到game_id → 尝试对应配置包 → 如果失败（可能版本不兼容） → 提示用户"检测到非标准版本，推荐使用Steam版获得最佳体验"
- 不强制要求正版，但明确告知："Steam版配置包成功率92%，本地导入版成功率67%"

### 8.3 存档问题——网吧最大的痛点在手机上的解法

**网吧的存档问题**：网吧重启后回写盘清空，存档丢失。所以网吧用户要么：
- 依赖游戏自带的云存档（Steam Cloud等）
- 用U盘拷走存档
- 接受存档丢失

**盖世的存档场景比网吧更复杂**：

| 场景 | 问题 | 用户痛感 |
|------|------|---------|
| Wine Prefix崩溃/重建 | 存档在Wine Prefix内，Prefix损坏=存档丢失 | ⭐⭐⭐ 致命 |
| 更换手机 | 存档在旧手机本地 | ⭐⭐ 高 |
| 配置包更新（Wine版本升级） | 新Prefix可能读不了旧存档 | ⭐⭐ 高 |
| 多设备游玩 | 手机玩了一半，想在PC继续 | ⭐ 有需求但不急 |
| 同一游戏换配置方案 | 切换配置后存档是否保留 | ⭐⭐ 高 |

### 8.4 存档方案设计

#### 层级一：本地存档保护（必做，第一期）

```
Wine Prefix 结构：
c:\users\z3635\...
├── drive_c/
│   ├── users/
│   │   └── 用户名/
│   │       ├── AppData/        ← 90%游戏存档在这里
│   │       ├── Documents/      ← 部分游戏存档在这里
│   │       └── Saved Games/    ← 部分游戏存档在这里
│   └── Program Files/
│       └── 游戏文件/
├── 系统注册表等配置文件
└── ...

盖世的存档保护机制：
┌──────────────────────────────────────────────────┐
│  启动游戏前：                                      │
│  自动快照 AppData/Documents/Saved Games → 备份区   │
│                                                    │
│  配置包更新/Prefix重建时：                          │
│  创建新Prefix → 迁移存档文件到新Prefix → 验证完整性 │
│                                                    │
│  崩溃恢复时：                                      │
│  从最近快照恢复存档 → 注入到修复后的Prefix          │
└──────────────────────────────────────────────────┘
```

**实现要点**：
- 配置包内定义该游戏的存档路径（每个游戏不一样）
- 存档路径写在配置包的 `save_paths` 字段里
- 启动前自动备份，确保Prefix崩了存档不丢

配置包扩展字段：
```json
{
  "save_config": {
    "save_paths": [
      "drive_c/users/{user}/AppData/Local/GTA V/",
      "drive_c/users/{user}/Documents/Rockstar Games/GTA V/Profiles/"
    ],
    "save_size_estimate_mb": 50,
    "cloud_save_supported": true,
    "cloud_save_platform": "steam_cloud"
  }
}
```

#### 层级二：云存档同步（第二期，配合VIP）

```
┌──────────────────────────────────────────┐
│  存档同步架构                              │
│                                            │
│  ┌─────────┐         ┌─────────────┐     │
│  │ 手机端   │ ──上传──→│ 盖世云存储    │     │
│  │ 本地存档  │ ←─下载──│ (对象存储)   │     │
│  └─────────┘         └──────┬──────┘     │
│                              │             │
│              不同设备同步：    │             │
│  ┌─────────┐                │             │
│  │ 另一台手机│ ←─────────────┘             │
│  └─────────┘                              │
│                                            │
│  触发同步的时机：                            │
│  - 退出游戏时自动上传（WiFi下）              │
│  - 启动游戏前自动拉取最新存档                │
│  - 手动触发"同步存档"                        │
└──────────────────────────────────────────┘
```

**与Steam Cloud的关系**：
| 场景 | 处理方式 |
|------|---------|
| 游戏支持Steam Cloud | 优先依赖Steam Cloud（已有成熟机制），盖世做二次备份 |
| 游戏不支持Steam Cloud | 盖世云存档是唯一保障 |
| 本地导入的游戏 | 只能靠盖世云存档 |
| Epic/其他平台 | 盖世云存档为主 |

#### 层级三：存档与配置包版本兼容（第二期）

配置包更新（Wine版本升级等）时，存档必须能继续用：

```
配置包 V1.0（Wine 9.5）→ 配置包 V2.0（Wine 9.8）

更新流程：
1. 检测用户本地存档
2. 备份存档到安全区
3. 创建新Wine Prefix（V2.0配置）
4. 迁移存档文件到新Prefix
5. 启动游戏验证存档可读
6. 如果失败 → 回退到V1.0配置 + 原存档，标记"此游戏暂不升级配置"
```

### 8.5 不同游戏来源的存档策略矩阵

| 游戏来源 | 存档位置 | 云同步依赖 | 盖世需要做什么 |
|---------|---------|-----------|-------------|
| Steam绑定 + 支持Steam Cloud | Steam Cloud + 本地 | Steam Cloud为主 | 本地快照备份（防Prefix崩溃） |
| Steam绑定 + 不支持Steam Cloud | 仅本地 | 无 | **盖世云存档（必须）** + 本地快照 |
| Epic绑定 | 仅本地（Epic Cloud覆盖率低） | 基本无 | **盖世云存档（必须）** + 本地快照 |
| 本地导入 | 仅本地 | 无 | **盖世云存档（必须）** + 本地快照 |
| CDKEY购买 | 取决于平台 | 取决于平台 | 盖世云存档兜底 |

### 8.6 存档作为留存壁垒

回到「差异化竞争策略」壁垒表的 #14（资产沉淀）——当前盖世是"零沉淀"。

**存档 = 用户在盖世平台的核心资产**。如果做好存档管理：
- 用户的存档在盖世云端 → 换到竞品=失去进度 → 迁移成本 → 留存壁垒
- 用户的配置+存档+按键方案都在盖世 → 完整的游戏资产沉淀

这恰好也对应网吧的留客逻辑：网吧虽然重启清档，但**网吧会员系统+云桌面**正是为了解决这个问题——让用户的数据留在这家网吧，下次还来这家。

## 九、关键风险和应对

| 风险 | 严重性 | 原因 | 应对 |
|------|--------|------|------|
| **安卓碎片化导致配置包不通用** | 高 | 同一芯片不同厂商的ROM有差异 | 芯片组+RAM+GPU三维分组，不按单机型 |
| **游戏更新后配置包失效** | 中 | Steam游戏更新可能改变兼容性 | 数据上报自动检测：成功率骤降→触发重测 |
| **冷启动期配置包覆盖不够** | 中 | 用户想玩的游戏没有配置包 | 优先覆盖热门游戏；无配置包的走传统模式（保留高级配置入口） |
| **本地导入游戏版本识别不准** | 中 | 魔改版/旧版/带MOD的游戏难以准确识别 | 多重识别策略（exe指纹+文件夹结构+用户确认）；标注"非标准版本成功率可能较低" |
| **存档迁移失败** | 中 | 配置包升级后新Prefix读不了旧存档 | 升级前强制备份；失败自动回退旧配置；存档路径写入配置包确保可追踪 |
| **云存档存储成本** | 低 | 存档文件通常不大（单游戏几十MB） | VIP用户不限量；免费用户限3个游戏云存档 |
| **云游戏兜底的成本** | 中 | 云GPU服务器贵 | 限制免费云游戏时长（如首次15分钟体验），VIP不限 |
| **配置包被竞品抄袭** | 低 | 配置参数本身不是秘密 | 壁垒不在单个配置包，在数据飞轮的规模和速度 |

## 十一、一句话总结

> **网吧的"点击就玩"完全可以在手机上实现**。不是搬网吧的硬件架构，而是搬网吧的**产品理念**：平台替用户做完所有配置，只把确定能玩的游戏放在用户面前。技术上靠"预配置库+数据飞轮+云游戏兜底"三件套，游戏来源靠"统一识别层"兼容Steam/Epic/本地导入，存档靠"本地快照+云存档"解决网吧模式下的数据持久化问题。
