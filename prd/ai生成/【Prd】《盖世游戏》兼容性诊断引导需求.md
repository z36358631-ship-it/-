## 二、背景与目标

**背景：** 用户在使用盖世游戏启动PC游戏时，黑屏、闪退/崩溃、卡在启动/加载是最高频的3类问题。用户遇到问题后缺乏引导，只能自行摸索或放弃。V1.0的逐项检测+逐项修复流程过重，核心解决方案实际是切换PC启动引擎方案。

**目标：** 客户端识别到启动异常后主动弹窗，引导用户一键切换引擎方案，同时收集用户遇到的问题类型帮助定位根因，降低因兼容性问题导致的用户流失率。

**核心挑战：**
- 3种异常场景的识别准确性（黑屏需区分有声音无画面 vs 完全无响应）
- 引擎方案推荐需按芯片+游戏动态匹配，展示启动成功率
- 问题类型标签需支持后台配置+多语言

## 三、故事介绍

### 3.1 用户与运营场景

**场景1：横屏游戏中触发**
用户启动艾尔登法环后出现黑屏（有声音没画面），客户端识别到异常，自动在游戏画面上弹出诊断弹窗，展示3个可选引擎方案和启动成功率。用户选择"系统智能设置方案"（成功率94.4%），可选标记"黑屏"标签并描述"有声音没画面"，点击"切换方案"→ 二次确认 → 应用成功，重启游戏后正常进入。

**场景2：竖屏详情页闪退后**
用户玩博德之门3时闪退，重新打开APP进入游戏详情页，系统检测到上次异常退出，弹出竖屏弹窗推荐切换引擎方案。用户选择方案后确认切换，重启游戏恢复正常。

### 3.2 价值分析

- 降低流失：将"遇到问题就卸载"转化为"一步切换方案即可解决"
- 提升启动成功率：引导用户切换到成功率最高的引擎方案
- 数据沉淀：收集用户反馈的问题类型+描述，反哺引擎方案优化
- 降低客服压力：用户自助切换，减少人工介入

### 3.3 核心体验路径

- C端用户路径：游戏启动异常 → 弹窗展示 → 选择引擎方案 → 可选标记问题类型+描述 → 切换方案 → 二次确认 → 应用成功 → 手动重启游戏
- B端运营路径：查看问题上报列表 → 分析反馈类型分布 → 配置/调整异常类型标签

### 3.4 产品指标预测

- 弹窗展示后方案切换率 ≥ 50%
- 切换后游戏启动成功率 ≥ 85%
- 问题类型反馈填写率 ≥ 30%（可选项）
- 用户因兼容性问题7日流失率降低 15%
- 客服兼容性相关工单量降低 30%

### 3.5 路径规划

- V2.0（当前）：主动弹窗+引擎方案切换+问题上报+后台文案配置
- V3.0（预告）：根据问题反馈数据自动优化引擎方案推荐权重；接入崩溃日志自动关联

#### 4.2.0 异常场景识别与触发规则

**三种检测场景：**

|场景|识别条件|典型表现|
|---|---|---|
|黑屏（最高频）|游戏进程存活但连续N秒无有效画面帧输出（视频流全黑/无渲染输出）|进去就黑屏、有声音没画面、改分辨率后黑屏、EA登录界面黑屏|
|闪退/崩溃|游戏进程非用户主动退出（进程异常终止/信号中断），心跳回溯检测到异常退出记录|启动即闪退、游戏中途闪退、UE崩溃报错、更新后闪退|
|卡在启动/加载|游戏启动后超过设定阈值（如3分钟）未进入可交互画面|Download Config失败、卡在检查环境界面、一直加载进不去、组件安装卡死|

**弹窗触发逻辑：**

|规则|说明|
|---|---|
|触发时机-黑屏|游戏运行中实时检测，连续10秒无有效画面帧 → 触发弹窗|
|触发时机-闪退|下次打开APP时，检测到上次该游戏进程异常退出记录 → 进入游戏详情页时触发弹窗|
|触发时机-卡加载|游戏启动后超过3分钟未进入可交互画面 → 触发弹窗|
|频控|同一游戏24小时内最多弹出2次，超过不再弹出|
|去重|同一次启动过程中只触发一次（如先卡加载再黑屏，只弹第一次识别到的场景）|
|冷却|用户点击"暂不处理"后，本次游戏会话不再弹出|
|前置条件|方案接口可用且返回≥1个可选方案时才弹窗；接口不可用且无本地缓存时不弹|

#### 4.2.1 引擎方案推荐规则

**匹配维度：** 同游戏 + 同GPU（芯片型号）

**方案推荐逻辑：**

|步骤|规则|
|---|---|
|①筛选|从推荐配置表中筛选出 当前游戏 + 当前设备GPU 维度下的所有可用引擎方案|
|②排除当前|排除用户当前已在使用的引擎方案（当前方案明显无效才触发弹窗）|
|③排序|按启动成功率降序排列|
|④截取|取前3个方案展示；不足3个则有几个展示几个（至少1个）|
|⑤默认选中|第1个（成功率最高）默认选中，标记"推荐"标签|

**启动成功率计算方式：**

|字段|定义|
|---|---|
|分子|使用该方案 + 同游戏 + 同GPU 维度下，成功启动并进入可交互画面的次数|
|分母|使用该方案 + 同游戏 + 同GPU 维度下，总启动次数|
|公式|成功率 = 成功启动次数 / 总启动次数 × 100%|
|统计周期|滚动30天|
|最小样本量|总启动次数 ≥ 50 时展示成功率数值；< 50 时显示"数据积累中"不展示具体百分比|
|数据来源|客户端埋点上报（每次启动成功/失败+当前引擎方案+游戏ID+GPU型号）|
|更新频率|T+1，每日凌晨聚合计算|

#### 4.2.1 边界条件-内容侧

|场景|处理方式|
|---|---|
|推荐配置表中无该芯片数据|展示通用全局方案（全芯片成功率最高的3个方案）|
|推荐配置表中无该游戏数据|展示该芯片维度全局成功率最高的3个方案|
|方案不足3个|有几个展示几个，至少展示1个|
|所有方案成功率相同|按方案创建时间倒序展示|
|异常类型标签被后台全部禁用|隐藏整个"启动异常类型"区域|
|网络异常无法获取方案数据|使用本地缓存的上次方案数据；若无缓存则弹窗不弹出|

#### 4.2.2 边界条件-用户侧

|场景|处理方式|
|---|---|
|用户点击"暂不处理"关闭弹窗|本次不再弹出，下次启动该游戏若仍异常则再次弹出|
|同一游戏多次触发|24小时内最多弹出2次|
|用户未选方案直接点"切换方案"|默认选中第1个方案，可直接提交|
|描述输入超过200字|禁止继续输入，显示字数统计"200/200"|
|弱网下提交反馈|本地缓存反馈数据，恢复网络后自动上报|
|切换方案后仍异常|下次弹出时不做特殊处理，同正常流程|

## 五、非功能需求

|需求类型|详细要求|
|---|---|
|性能|弹窗渲染 ≤ 200ms；方案数据接口响应 ≤ 500ms|
|兼容性|Android 8.0+；iOS 14+；支持骁龙8系列、天玑9000系列主流芯片|
|安全|上报数据不包含用户账号/密码；描述文字需敏感词过滤（提交时）|
|容错|方案接口超时降级为本地缓存；反馈提交失败本地暂存，恢复网络后重试|
|频控|同一游戏24小时内最多自动弹出2次|

## 七、运营需求

1. **引擎方案维护：** 按芯片+游戏维度持续更新推荐引擎方案数据及启动成功率
2. **异常类型标签配置：** 上线前完成国内版（中文）和海外版（中+英）的标签配置
3. **上报问题跟进：** 每日处理待处理上报，根据反馈类型分布调整引擎方案推荐优先级
4. **数据分析：** 定期分析"切换方案后仍失败"的case，推动引擎侧优化
